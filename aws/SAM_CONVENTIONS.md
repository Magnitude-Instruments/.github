# AWS SAM Conventions

Project structure, naming, and deployment conventions for Magnitude Instruments SAM projects.

## Project Structure

```
project-root/
├── template.yaml              # SAM template (all AWS resources)
├── samconfig.toml             # Deployment config (auto-generated by sam deploy --guided)
└── lambdas/
    ├── function_name/
    │   ├── app.py             # Lambda handler
    │   └── requirements.txt   # Dependencies (omit if only using boto3)
    └── another_function/
        └── app.py
```

- One `template.yaml` per logical group of related functions (e.g., customer portal API, cognito triggers, support automation)
- Lambda code lives in `lambdas/<function_name>/app.py`
- Shared code goes in a Lambda layer under `layers/`

## Naming

### CloudFormation Stack Names

Descriptive, kebab-case: `customer-portal-backend`, `maginsts-support-automation`

### Lambda Function Names

Prefix with an abbreviation of the project, kebab-case:

| Project | Prefix | Example |
|---------|--------|---------|
| Customer Portal API | `cp-` | `cp-list-all-files`, `cp-get-presigned-url` |
| Customer Portal Cognito | `notify-` | `notify-admin-signup`, `notify-user-confirmed` |
| Support Automation | `support-` | `support-process-new-ticket`, `support-update-ticket-age` |

### SAM Logical IDs

PascalCase, descriptive:

```yaml
ListAllFilesFunction:        # Lambda function
CustomerPortalApi:           # API Gateway
TicketEventsTable:           # DynamoDB table
PublicAssetsBucket:          # S3 bucket
```

## Template Structure

Order resources consistently:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Transform: AWS::Serverless-2016-10-31
Description: >
  Brief description of what this stack does.

Globals:
  Function:
    Runtime: python3.14
    MemorySize: 128
    Architectures:
      - x86_64

Resources:
  # 1. API Gateway (if applicable)
  # 2. Lambda Functions
  # 3. Lambda Permissions (Cognito invoke, etc.)
  # 4. DynamoDB Tables
  # 5. S3 Buckets
  # 6. Other resources

Outputs:
  # API URLs, function ARNs, bucket names
```

## API Gateway

### REST API with Cognito Authorizer

```yaml
CustomerPortalApi:
  Type: AWS::Serverless::Api
  Properties:
    Name: customer-portal-api
    StageName: prod
    Cors:
      AllowMethods: "'GET,OPTIONS'"
      AllowHeaders: "'Content-Type,X-Amz-Date,Authorization,X-Api-Key,X-Amz-Security-Token'"
      AllowOrigin: "'*'"
    Auth:
      DefaultAuthorizer: CognitoAuthorizer
      AddDefaultAuthorizerToCorsPreflight: false    # OPTIONS must be unauthenticated
      Authorizers:
        CognitoAuthorizer:
          UserPoolArn: !Sub arn:aws:cognito-idp:${AWS::Region}:${AWS::AccountId}:userpool/<pool-id>
    GatewayResponses:                               # CORS headers on error responses
      DEFAULT_4XX:
        ResponseParameters:
          Headers:
            Access-Control-Allow-Origin: "'*'"
            Access-Control-Allow-Headers: "'Content-Type,Authorization'"
            Access-Control-Allow-Methods: "'GET,OPTIONS'"
      DEFAULT_5XX:
        ResponseParameters:
          Headers:
            Access-Control-Allow-Origin: "'*'"
            Access-Control-Allow-Headers: "'Content-Type,Authorization'"
            Access-Control-Allow-Methods: "'GET,OPTIONS'"
```

Always include `GatewayResponses` for 4XX/5XX so the browser doesn't mask auth errors as CORS errors.

### Stage Naming

Use `prod` for all SAM API stages. Environment separation is handled at the stack level, not the stage level.

## IAM Policies

Scope permissions to the minimum required. Never use broad AWS managed policies like `AmazonS3ReadOnlyAccess`.

**SAM policy templates** (e.g., `DynamoDBCrudPolicy`, `AWSSecretsManagerGetSecretValuePolicy`) are acceptable — they generate scoped inline policies under the hood and are a safe convenience.

For S3 or other cases not covered by a SAM policy template, use inline statements scoped to specific resources:

```yaml
Policies:
  - DynamoDBCrudPolicy:                          # SAM policy template — OK
      TableName: !Ref TicketEventsTable
  - AWSSecretsManagerGetSecretValuePolicy:       # SAM policy template — OK
      SecretArn: !Sub arn:aws:secretsmanager:${AWS::Region}:${AWS::AccountId}:secret:magnitude/asana-pat-*
  - Statement:                                    # Inline statement for S3
      - Effect: Allow
        Action: s3:ListBucket
        Resource: arn:aws:s3:::maginstsrepobucket
  - Statement:
      - Effect: Allow
        Action: s3:GetObject
        Resource: arn:aws:s3:::maginstsrepobucket/*
```

Separate `ListBucket` (on the bucket ARN) from `GetObject` (on the object ARN with `/*`).

## Environment Variables

Use environment variables for values that vary between environments or contain secrets:
- Bucket names
- Table names
- Email addresses
- Secret ARNs
- URLs
- `LOG_LEVEL` (default `INFO` — allows switching to `DEBUG` without redeploying)

Static identifiers that never change across environments (e.g., third-party API field IDs) may remain as module-level constants in the Lambda code (see PYTHON_LAMBDA_STYLE.md).

```yaml
Environment:
  Variables:
    S3_BUCKET: maginstsrepobucket
    SENDER_EMAIL: "Magnitude Instruments <noreply@magnitudeinstruments.com>"
    SECRET_NAME: magnitude/asana-pat
```

Reference dynamic resources with `!Ref` or `!Sub` when possible:

```yaml
TICKET_EVENTS_TABLE: !Ref TicketEventsTable
LOGO_URL: !Sub https://${PublicAssetsBucket}.s3.amazonaws.com/logos/logo.png
```

## Deployment

### Build

Always build in a container to ensure Lambda-compatible binaries:

```bash
sam build --use-container
```

### First Deploy

```bash
sam deploy --guided
```

Suggested settings:
- **Confirm changes before deploy**: Yes
- **Allow SAM CLI IAM role creation**: Yes
- **Disable rollback**: No
- **Save arguments to samconfig.toml**: Yes

### Subsequent Deploys

```bash
sam deploy
```

### What Not to Commit

Add to `.gitignore`:

```
**/.aws-sam/
```

The `samconfig.toml` may be committed if it contains no secrets (it typically doesn't), but deployment-specific configs (account-specific bucket names) should be in `.gitignore` if they vary between developers.

## Lambda Timeouts

The default Lambda timeout is 3 seconds, which is too short for functions that call external APIs (Asana, SES, Google Chat, etc.). Set an explicit `Timeout` in the SAM template:

```yaml
Globals:
  Function:
    Timeout: 30       # 30 seconds — reasonable default for API-calling Lambdas

# Or per-function for longer operations:
SlowFunction:
  Properties:
    Timeout: 120      # 2 minutes for batch/scheduled tasks
```

- **API-backed functions**: 30 seconds is a safe default
- **Scheduled/batch functions**: Set based on expected workload, up to 900 seconds (15 minutes max)

## Local Development

Use SAM CLI for local testing before deploying:

```bash
# Invoke a single function with a test event
sam local invoke ProcessNewTicketFunction -e events/test_event.json

# Start a local API Gateway
sam local start-api
```

Store sample events in an `events/` directory at the project root for repeatable local testing.

## Region

All resources deploy to **us-east-1**. Hardcode the region in boto3 clients:

```python
boto3.client("s3", region_name="us-east-1")
```
